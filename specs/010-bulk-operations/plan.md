# Implementation Plan: Bulk Operations

**Branch**: `010-bulk-operations` | **Date**: 2026-03-18 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/010-bulk-operations/spec.md`

## Summary

Implement 6 MCP tools for bulk task and section operations in OmniFocus: `move_tasks`, `duplicate_tasks`, `batch_update_tasks`, `convert_tasks_to_projects`, `move_sections`, and `duplicate_sections`. All tools follow the established definitions/primitives/contracts architecture with per-item success/failure results (Phase 5/13 batch pattern). OmniJS bulk API functions (`moveTasks()`, `duplicateTasks()`, `convertTasksToProjects()`, `moveSections()`, `duplicateSections()`) are called one item at a time within a single script execution to maintain per-item error correlation. `batch_update_tasks` is a "wide" tool accepting any combination of updatable task properties applied uniformly across 1-100 tasks.

## Architecture Decisions

### AD-01: One-item-at-a-time OmniJS execution
Call `moveTasks([task], position)`, `duplicateTasks([task], position)`, etc. one item at a time within a single OmniJS script. This maintains 1:1 input-output correlation for per-item results (FR-007) while keeping a single process spawn for performance.

### AD-02: Per-domain contract ownership
Bulk-tools owns its own `ItemIdentifierSchema`, `BulkBatchItemResultSchema`, and `SummarySchema` rather than importing from status-tools. Follows the established per-domain pattern documented in existing disambiguation.ts files.

### AD-03: Two position schemas
`TaskPositionSchema` (project/task/inbox targets) and `SectionPositionSchema` (folder/library root targets) are separate schemas because they accept fundamentally different target types per the OmniFocus API.

### AD-04: Flat PropertyUpdateSet with explicit clearX flags
`batch_update_tasks` uses a flat object with `clearDueDate`, `clearDeferDate`, `clearEstimatedMinutes`, `clearPlannedDate` boolean flags. This avoids null/undefined ambiguity and matches the existing pattern in the codebase.

### AD-05: Target pre-validation before item loop
All tools validate the shared target location (project, folder, inbox) before iterating over items (FR-016). Invalid target returns a single top-level error rather than N identical per-item errors.

### AD-06: Duplicate copies always active/incomplete
`duplicate_tasks` explicitly calls `markIncomplete()` on copies after duplication (FR-011). This ensures template-workflow use cases produce usable copies.

### AD-07: convert_tasks_to_projects uses convertTasksToProjects() API
Uses the native OmniJS `convertTasksToProjects()` function rather than manually creating projects and moving subtasks. Called one item at a time for per-item error handling.

### AD-08: Error code taxonomy
9 error codes covering all failure modes: NOT_FOUND, DISAMBIGUATION_REQUIRED, TARGET_NOT_FOUND, OPERATION_FAILED, TAG_NOT_FOUND, RELATIVE_TARGET_NOT_FOUND, ALREADY_A_PROJECT, VERSION_NOT_SUPPORTED, VALIDATION_ERROR.
