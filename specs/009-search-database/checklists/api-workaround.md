# API Workaround Checklist: Search & Database (SPEC-009)

**Purpose**: Validate requirements quality for OmniJS API workarounds, limitations, and assumptions -- specifically `flattenedTasks` iteration performance, `canUndo`/`canRedo` pre-checks, `save()`/`cleanUp()` behavior documentation, and large-collection iteration concerns
**Created**: 2026-03-18
**Feature**: [spec.md](../spec.md) | [research.md](../research.md) | [quickstart.md](../quickstart.md)
**Domain**: api-workaround
**Depth**: Standard
**Audience**: Reviewer (PR)

## Requirement Completeness -- flattenedTasks Iteration for Search

- [ ] CHK001 - Is the absence of a `tasksMatching()` API documented as a constraint in the spec itself, or only in research/plan artifacts? [Completeness, Spec Assumptions] -- The spec assumes `flattenedTasks.filter()` is the search mechanism but does not state WHY this workaround is necessary. A reviewer unfamiliar with OmniJS would not understand the design rationale from the spec alone.
- [ ] CHK002 - Are requirements defined for how `flattenedTasks` handles the root task of each project (the synthetic task that represents the project)? [Gap] -- The quickstart and plan document a "root task skip" pattern, but the spec has no requirement or edge case addressing root tasks appearing in search results or statistics. This is an implicit API workaround with no requirements-level documentation.
- [ ] CHK003 - Is the behavioral difference between `flattenedTasks.filter()` (database order) and `*Matching()` (relevance order) specified as a user-facing requirement? [Clarity, Spec Assumptions] -- The spec acknowledges this asymmetry in Assumptions but does not surface it in functional requirements or acceptance scenarios. Should users expect different result ordering between task search and project/folder/tag search?
- [ ] CHK004 - Are requirements defined for what `flattenedTasks` includes beyond active tasks (completed, dropped, inbox items, task groups)? [Completeness, Spec Assumptions] -- The spec states `flattenedTasks` includes ALL tasks, but FR-006 only defines the status filter. Are task group items (parent tasks of task hierarchies) expected in search results, or should they be filtered like root tasks?
- [ ] CHK005 - Is the case-insensitive substring matching strategy for task search specified as a user-visible behavioral contract? [Clarity, Spec §FR-001] -- FR-001 says "case-insensitive substring matching" but does not specify whether this is a requirement or an implementation detail. If a future `tasksMatching()` API emerges with different matching behavior, would that violate the spec?

## Requirement Completeness -- Performance of Large Collection Iteration

- [ ] CHK006 - Is the "under 10,000 items" database size assumption specified as a performance boundary with defined degradation behavior? [Measurability, Spec Assumptions] -- The spec assumes acceptable performance for typical databases but does not define what happens at 10,000+ items: timeout, truncation, or error. Is there a requirement for a performance ceiling?
- [ ] CHK007 - Are performance requirements specified for `get_database_stats`, which iterates both `flattenedTasks` AND `flattenedProjects` in a single script? [Gap, Spec §FR-014] -- Stats computation iterates two full collections plus accesses `flattenedFolders.length`, `flattenedTags.length`, and `inbox.length`. No performance target or timeout is defined for this multi-collection operation.
- [ ] CHK008 - Is the combined status-filter-and-name-match pattern for `search_tasks` documented as a performance-motivated design decision? [Clarity] -- The quickstart applies both name matching and status filtering within the same `flattenedTasks.filter()` callback. The spec does not explain whether this is a performance optimization (single pass) or whether separate filtering passes would be acceptable.
- [ ] CHK009 - Are requirements defined for the behavior when `flattenedTasks` returns an extremely large collection (e.g., 50,000+ items) and the filter callback is slow? [Gap, Edge Case] -- No timeout, circuit breaker, or maximum iteration count is specified. The spec's SC-001 says "same response time as existing item-listing operations" but does not define an absolute performance budget.
- [ ] CHK010 - Is the performance difference between `inbox.length` (property access) and `flattenedTasks.forEach()` (full iteration) documented as a requirements-level distinction? [Clarity, Spec §FR-015] -- FR-015 specifies a "lightweight" inbox count. Is "lightweight" defined relative to other operations, or is it an absolute performance requirement?

## Requirement Completeness -- canUndo/canRedo Pre-Checks

- [ ] CHK011 - Is the `canUndo`/`canRedo` being properties (not methods) specified in the requirements or only in research artifacts? [Gap, Spec Assumptions] -- The spec states these are "read-only Boolean properties (not method calls)" in Assumptions, but this is an implementation constraint, not a user-facing requirement. If a developer misreads this as methods, the OmniJS will fail silently.
- [ ] CHK012 - Are requirements defined for the atomicity of the `canUndo` check followed by `undo()` call? [Gap, Spec §FR-010] -- The pre-check and execution happen in two separate statements within the OmniJS script. Is there a race condition requirement if OmniFocus modifies the undo stack between the check and the call? Is this documented as safe because OmniJS runs synchronously?
- [ ] CHK013 - Is the `performed` field in the undo/redo response (true if executed, false if stack empty) documented as a user-facing requirement? [Completeness, Spec §FR-012] -- FR-012 says "clear, non-error message" but does not define the `performed` boolean field that distinguishes "nothing to undo" from "undo succeeded." The data model defines it, but is it a spec-level requirement?
- [ ] CHK014 - Are requirements defined for what happens if `undo()` or `redo()` throws despite the pre-check passing (edge case where stack changes between check and call)? [Coverage, Edge Case] -- The spec requires pre-checks but does not specify error handling if the method throws unexpectedly. Should the response be a success=false error, or is this scenario considered impossible?
- [ ] CHK015 - Is the scope of undo/redo (entire OmniFocus database, not MCP-scoped) documented as a requirement in the acceptance scenarios? [Clarity, Spec §US-9.5] -- Acceptance scenario 5 mentions the tool description should warn about database-wide scope, but no functional requirement explicitly states this scope constraint. Is FR-010 clear that undo reverses ANY recent change, not just MCP operations?

## Requirement Clarity -- save() and cleanUp() Behavior

- [ ] CHK016 - Is the synchronous behavior of `save()` specified as a requirement or only an assumption? [Clarity, Spec Assumptions] -- The spec assumes "save() is synchronous" but the research does not confirm whether this is synchronous or returns a Promise. If async, the OmniJS script pattern would need to change. Is this assumption validated?
- [ ] CHK017 - Is the void return of `cleanUp()` documented as a user-facing limitation? [Completeness, Spec §Clarifications] -- The clarification states `cleanUp()` returns no information about processed items, but is this limitation surfaced in the tool description or acceptance scenarios? Users may expect to know how many items were moved.
- [ ] CHK018 - Are requirements defined for what `cleanUp()` actually does beyond "processes inbox items"? [Clarity, Spec §FR-009] -- FR-009 says "triggers OmniFocus's built-in clean up behavior" but the research reveals cleanUp() also performs "delayed filtering" and "deletes empty items." Are these additional behaviors documented in requirements?
- [ ] CHK019 - Is the sync-triggering behavior of `save()` specified as a conditional requirement? [Clarity, Spec §FR-013] -- FR-013 says "triggers synchronization if enabled" but does not define: (a) how users know if sync is enabled, (b) whether the tool response should indicate sync status, or (c) what happens if sync fails but save succeeds.
- [ ] CHK020 - Are idempotency requirements for `save()` and `cleanUp()` defined with specific acceptance criteria? [Measurability, Spec §SC-007] -- SC-007 says "no adverse effects" when triggered on an already-saved/clean database, but no acceptance scenario defines what "no adverse effects" means measurably. Is success-true the only criterion?

## Requirement Consistency -- Top-Level vs Document-Level API Calls

- [ ] CHK021 - Is the requirement that database functions are top-level (not `document.*`) consistently stated across all relevant artifacts? [Consistency, Spec Assumptions] -- The spec says "`cleanUp()`, `save()`, `undo()`, and `redo()` are top-level Database functions in OmniJS context (not `document.*` methods)" but this conflicts with common OmniJS documentation patterns that show `document.cleanUp()`. Is this ambiguity resolved?
- [ ] CHK022 - Are the top-level function names (`save`, `cleanUp`, `undo`, `redo`, `canUndo`, `canRedo`) and their exact casing documented as API surface requirements? [Clarity] -- The spec uses `cleanUp()` (camelCase) consistently, but no requirement explicitly defines the exact function names and casing. A typo (e.g., `cleanup()` vs `cleanUp()`) would cause a silent failure in OmniJS.

## Edge Case Coverage -- API Workaround Boundaries

- [ ] CHK023 - Are requirements defined for what happens when `flattenedTasks` is empty (new OmniFocus installation with no tasks)? [Coverage, Edge Case] -- The spec's edge case section addresses "empty database" for statistics but not explicitly for search. Does FR-007 (empty result set, not error) cover this case, or should it be explicit?
- [ ] CHK024 - Are requirements defined for how `*Matching()` handles special characters in the query string (quotes, brackets, regex metacharacters)? [Gap, Spec §FR-016] -- FR-016 requires non-empty query but does not address special character handling. Smart Match may interpret certain characters differently from substring matching, creating inconsistent behavior across search tools.
- [ ] CHK025 - Are requirements defined for the behavior when `inbox.length` is called but the inbox collection is not yet loaded or OmniFocus is still initializing? [Gap, Edge Case] -- The spec assumes inbox access always works but does not address startup race conditions where OmniFocus may not have fully loaded its database.
- [ ] CHK026 - Is the search query injection risk documented for the `flattenedTasks.filter()` workaround? [Gap, Edge Case] -- The quickstart shows template-literal interpolation (`var query = "${query}";`) for the search query. Are requirements defined for sanitizing the query string to prevent OmniJS code injection via crafted search terms?

## Dependency and Assumption Validation

- [ ] CHK027 - Is the assumption that `Task.Status` enum has exactly 7 values (Available, Blocked, Completed, Dropped, DueSoon, Next, Overdue) validated against a specific OmniFocus version? [Assumption, Spec Assumptions] -- The status mapping depends on this exact enum. If a future OmniFocus version adds a new status value, the "active" filter would silently exclude those tasks. Is this version-pinned?
- [ ] CHK028 - Is the assumption that Smart Match behavior matches Quick Open behavior documented as version-specific? [Assumption, Spec Assumptions] -- The spec says `*Matching()` uses "Quick Open-style relevance ranking" but does not specify which OmniFocus version this applies to. Could the ranking algorithm change between versions?
- [ ] CHK029 - Is the assumption that `canUndo`/`canRedo` state does not change between the property read and the method call documented as relying on OmniJS synchronous execution? [Assumption, Spec Assumptions] -- The spec assumes pre-check safety but does not explicitly document that OmniJS scripts run synchronously (single-threaded), which is the prerequisite for this pattern to be safe.
- [ ] CHK030 - Are requirements defined for the OmniFocus minimum version required for the search and database APIs? [Gap, Dependency] -- The spec has no version constraint. Other specs (SPEC-007, SPEC-013) explicitly gate features on v4.7+ or v3.8+. Are `projectsMatching()`, `foldersMatching()`, `tagsMatching()`, and `canUndo`/`canRedo` available in all OmniFocus Pro versions with Omni Automation support?

## Requirement Completeness -- flattenedProjects/flattenedFolders/flattenedTags Iteration for Search

- [ ] CHK031 - Are requirements defined for how `flattenedProjects`, `flattenedFolders`, and `flattenedTags` collections behave when their respective `*Matching()` methods are used instead of manual iteration? [Completeness, Gap] -- The spec documents `flattenedTasks.filter()` as a workaround but does not clarify whether `projectsMatching()`, `foldersMatching()`, and `tagsMatching()` internally iterate these flattened collections or use an optimized index. This affects performance expectations for large databases.
- [ ] CHK032 - Are requirements defined for error handling when `projectsMatching()`, `foldersMatching()`, or `tagsMatching()` receive an unexpected input (e.g., null, undefined, numeric)? [Gap, Edge Case] -- FR-016 requires non-empty string validation, but the spec does not address what Smart Match methods return for edge-case inputs that pass Zod validation but may trigger unexpected behavior in OmniJS (e.g., very long strings, Unicode-heavy queries).
- [ ] CHK033 - Is the complete set of item types returned by `flattenedTasks` documented as a requirements-level constraint? [Completeness, Spec Assumptions] -- The quickstart states `flattenedTasks` returns "active, completed, dropped, inbox items, root tasks for projects, and task groups" but the spec only mentions completed, dropped, and inbox items in Assumptions. Task groups (parent tasks of sub-task hierarchies) are not mentioned in the spec at all. Are they expected in search results?
- [ ] CHK034 - Are requirements defined for how `flattenedFolders` and `flattenedTags` handle nested hierarchies? [Clarity, Gap] -- `flattenedFolders` presumably flattens the folder tree, and `flattenedTags` flattens the tag hierarchy. The spec does not state whether `foldersMatching()`/`tagsMatching()` search the entire flattened hierarchy or only top-level items. Is full-hierarchy search a requirement or an assumption?
- [ ] CHK035 - Is the memory allocation behavior of `flattenedTasks.filter()` (which creates a new array of all matching tasks before applying `.slice()`) documented as a performance consideration? [Gap, Spec Assumptions] -- The quickstart pattern calls `flattenedTasks.filter()` (creates full match array), then `.slice(0, limit)`. For a database with 50,000 tasks where 10,000 match, this allocates a 10,000-element array only to return 50. No requirement addresses whether early termination or streaming is needed.

## Requirement Clarity -- document.save() vs Top-Level save() Ambiguity

- [ ] CHK036 - Is the distinction between the top-level `save()` function and `document.save()` method explicitly documented as a requirements constraint? [Clarity, Spec Assumptions] -- The Omni Automation API defines BOTH a top-level `save()` (Database class) and a `document.save()` (Document class). The spec assumes the top-level form, but research confirms both exist. Are requirements clear about which one to use and why they differ? The Document-level `save()` description says simply "Save this document" while the Database-level `save()` adds "triggers a sync request."
- [ ] CHK037 - Is the return type of the top-level `save()` documented as void (synchronous) with evidence? [Clarity, Spec Assumptions] -- The spec assumes `save()` is synchronous, and the API signature shows `save()` returns `()` (void). However, the Document class has a `makeNewAndShow()` that returns a `Promise`, suggesting the framework supports async operations. Is the assumption that `save()` is synchronous validated against actual runtime behavior, or only inferred from the API signature?
- [ ] CHK038 - Are requirements defined for what happens if `save()` is called while OmniFocus is mid-sync with a remote server? [Gap, Edge Case, Spec §US-7] -- The edge case section mentions "save triggered while OmniFocus is performing a sync" and says "the operation should complete normally," but this is stated as an edge case observation, not a functional requirement. Is there an FR that addresses concurrent save+sync behavior?

## Requirement Clarity -- cleanUp() Detailed Behavior and Side Effects

- [ ] CHK039 - Are the three distinct operations performed by `cleanUp()` documented as user-facing behavior in the spec? [Completeness, Spec §FR-009] -- The API documentation states `cleanUp()` performs three actions: (1) processes inbox items with assigned containers, (2) performs delayed filtering, and (3) deletes empty items. FR-009 and US-8 only mention inbox item processing. The "delayed filtering" and "deletes empty items" behaviors are undocumented in the spec. Could deletion of empty items surprise users?
- [ ] CHK040 - Is "deletes empty items" behavior of `cleanUp()` defined as a requirement with explicit scope? [Gap, Spec §FR-009] -- The API docs confirm `cleanUp()` "deletes empty items." The spec does not mention this destructive side effect. What constitutes an "empty item" in OmniFocus? Is this a task with no name? A project with no tasks? Users triggering cleanup via AI assistant may not expect item deletion.
- [ ] CHK041 - Are requirements defined for what "delayed filtering" means in the `cleanUp()` context? [Ambiguity, Spec §FR-009] -- The API documentation mentions `cleanUp()` "performs any delayed filtering" but neither the spec nor the research explains what "delayed filtering" is. The tag documentation mentions "Recent changes may not be reflected until the cleanUp() function is called." Is this filtering behavior documented as a requirement?
- [ ] CHK042 - Is the relationship between `cleanUp()` and `tag.availableTasks`/`tag.remainingTasks` accuracy documented? [Gap, Dependency] -- The Tag API docs state "Recent changes may not be reflected until the cleanUp() function of the Database class is called on the database" for both `availableTasks` and `remainingTasks` properties. Are requirements defined for when users should call `cleanup_database` to ensure accurate tag-task associations?

## Scenario Coverage -- Interaction Between Database Operations

- [ ] CHK043 - Are requirements defined for the interaction between `cleanUp()` and the undo stack? [Gap, Scenario Coverage] -- If `cleanUp()` moves inbox items to projects and deletes empty items, does this create undo-able operations? Can a user call `undo` after `cleanup_database` to reverse the cleanup? The spec does not address whether cleanup operations are undoable.
- [ ] CHK044 - Are requirements defined for the interaction between `save()` and the undo stack? [Gap, Scenario Coverage] -- Does calling `save()` clear or affect the undo stack? In many applications, saving is a checkpoint that clears undo history. If this is the case in OmniFocus, calling `save_database` followed by `undo` might fail unexpectedly. The spec does not address this interaction.
- [ ] CHK045 - Are requirements defined for the sequence: `undo` then immediately `save_database`? [Gap, Scenario Coverage] -- If a user undoes an operation via `undo` and then triggers `save_database`, is the undone state persisted and synced? The spec treats these as independent operations but does not address their composition.

## Non-Functional Requirements -- OmniJS Execution Model

- [ ] CHK046 - Is the synchronous, single-threaded execution model of OmniJS scripts documented as a requirement prerequisite? [Gap, Assumption] -- Multiple spec assumptions depend on OmniJS running synchronously (canUndo check + undo() atomicity, filter + slice sequencing, stats iteration consistency). No requirement or assumption explicitly states that OmniJS scripts run in a single-threaded synchronous context. This is the foundational assumption that makes the pre-check pattern safe.
- [ ] CHK047 - Are requirements defined for the maximum OmniJS script execution time before OmniFocus terminates or times out the script? [Gap, Non-Functional] -- `get_database_stats` iterates `flattenedTasks`, `flattenedProjects`, and accesses three `.length` properties. `search_tasks` iterates `flattenedTasks` with a filter. No requirement defines what happens if OmniFocus imposes an execution timeout on automation scripts. Is there a known timeout limit?
- [ ] CHK048 - Is the query string sanitization requirement for template literal interpolation documented for ALL search tools, not just `search_tasks`? [Consistency, Gap, Spec §FR-016] -- CHK026 identified injection risk for `flattenedTasks.filter()`, but `projectsMatching()`, `foldersMatching()`, and `tagsMatching()` also receive the query via template interpolation (`var query = "${query}"`). Are sanitization requirements defined consistently across all four search tools?
