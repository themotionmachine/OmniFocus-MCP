# Type Safety Checklist: Repetition Rule Management

**Purpose**: Validate completeness, clarity, and consistency of type safety requirements for the 5 repetition rule MCP tools, with special attention to opaque ICS strings, OmniJS↔TypeScript enum mappings, and Zod schema design.

## Opaque String Type Requirements

- [x] CHK001 - Is the ICS `ruleString` explicitly specified as an opaque pass-through type with NO format validation (only `z.string().min(1)`)? [Clarity, Spec §FR-002, Spec §Scope Boundaries] — spec.md §FR-002: "passing it through to OmniFocus without validation." §Scope Boundaries: ICS parsing explicitly out of scope. data-model.md: `z.string().min(1)` only.
- [x] CHK002 - Is the rationale for treating ICS strings as opaque documented — OmniFocus is the validation authority, not the MCP server? [Clarity, Spec §Assumptions] — spec.md §Assumptions: "we treat it as opaque — OmniFocus is the authority on what is valid." §Scope Boundaries §Out of Scope: "Parsing or validating ICS rule strings (pass-through to OmniFocus)."
- [x] CHK003 - Is it specified that ICS rule string validation errors are surfaced via OmniJS constructor exceptions (not Zod schema validation)? [Completeness, Spec §Error Handling Gap Resolution] — spec.md §Clarifications (Error Handling): "set_repetition — NOT_FOUND, invalid ICS string (OmniJS constructor throws)." §US-2 Scenario 4: "I receive an error from OmniFocus (we do not pre-validate ICS strings)."
- [x] CHK004 - Is the ICS string format described with sufficient examples for consumers to understand the expected shape (e.g., `FREQ=WEEKLY;BYDAY=MO`)? [Clarity, Spec §Key Entities, Data Model §PresetName] — spec.md §Key Entities: "e.g., FREQ=WEEKLY;BYDAY=MO." Acceptance scenarios include FREQ=DAILY, FREQ=MONTHLY;BYMONTHDAY=1, FREQ=WEEKLY;BYDAY=MO,WE,FR. data-model.md and research.md §Confirmed RRULE Components: component table with examples.
- [x] CHK005 - Is the `ruleString` return type in get_repetition response specified as `z.string()` (always present when hasRule is true, never validated beyond non-empty)? [Completeness, Data Model §RepetitionRuleData] — data-model.md §RepetitionRuleData: `ruleString | string | z.string() | ICS RRULE format`. Always present in RepetitionRuleData (which only exists when hasRule is true).

## String Embedding Safety

- [x] CHK027 - Is it documented that ICS rule strings must be escaped via `escapeForJS()` before embedding in OmniJS script templates? [Completeness, Plan §Reusable Patterns] — plan.md §Reusable Patterns: `escapeForJS()` listed as reusable pattern. spec.md §Type Safety Gap Resolution: details on escapeForJS for repetition primitives. quickstart.md uses `${escapedId}` and `${escapedRuleString}` showing escaped values.
- [x] CHK028 - Is the `escapeForJS()` pattern specified for the repetition tool primitives (matching the existing per-primitive duplication pattern)? [Clarity, ~~Gap~~ Resolved] — Follow per-primitive duplication pattern (26+ existing copies). Define `escapeForJS()` locally in each repetition primitive. Not shared — this is the established codebase convention. Documented in spec.md §Type Safety Gap Resolution.
- [x] CHK029 - Are the characters escaped by `escapeForJS()` documented (`\ " \n \r \t`) for the repetition string types? [Clarity, ~~Gap~~ Resolved] — Escapes 5 characters in order: `\` first (avoid double-escaping), then `"`, `\n`, `\r`, `\t`. Standard ICS strings contain none of these, but user-provided ruleString must still be escaped. Documented in spec.md §Type Safety Gap Resolution.
- [x] CHK030 - Is it specified that standard ICS rule strings (e.g., `FREQ=WEEKLY;BYDAY=MO`) contain no characters requiring escaping, but user-provided arbitrary strings might? [Edge Case] — spec.md §Type Safety Gap Resolution: "Standard ICS rule strings (FREQ=WEEKLY;BYDAY=MO) contain no special characters, but user-provided ruleString input must still be escaped for safety."
